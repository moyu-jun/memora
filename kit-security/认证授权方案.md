# 认证授权方案

> 适用于中高并发业务系统的 Opaque Token 认证授权设计。
>
> 目标是：**高频鉴权链路只做 1 次 Redis 读取**，同时保留多端登录、踢下线、权限变更、Refresh Token 续期等控制能力。

## 1. 文档信息

- **文档状态**：Draft
- **适用范围**：企业内部系统、后台管理系统、API 网关 + 后端服务架构
- **核心目标**：低延迟鉴权、可控失效、可维护的权限管理

---

## 2. 设计背景

传统方案通常有两类：

1. **JWT 本地验签**
   - 优点：性能高、资源服务可独立校验
   - 缺点：踢下线、权限变更、会话管理的即时性较弱

2. **Opaque Token + 每次请求回源校验**
   - 优点：控制能力强
   - 缺点：热路径容易产生多次 Redis 往返，放大延迟和压力

本方案取两者之间的平衡：

- 对外使用 **Opaque Token**
- 热路径只依赖 **`auth:access:{token}`**
- 低频控制面负责会话、刷新、踢下线、权限变更
- 权限集合通过**本地缓存**或**短期缓存**承接

---

## 3. 设计目标

### 3.1 核心目标

- 使用 Opaque Token 作为访问凭证，不直接暴露用户状态和权限细节。
- 支持多端登录、Refresh Token 续期、主动退出、踢下线、权限变更。
- **高频鉴权链路尽量只做 1 次 Redis 读取。**
- 在性能、可维护性、权限变更生效速度之间取得平衡。

### 3.2 设计原则

1. **高频路径极简化**  
   每个资源请求只做一次主查询，优先读取 `auth:access:{token}`。

2. **控制面与数据面分离**  
   登录、刷新、退出、踢下线、权限变更属于低频控制面；资源请求鉴权属于高频数据面。

3. **权限缓存化**  
   权限集合不作为每个请求的 Redis 强依赖，而是通过本地缓存或短期缓存承接。

4. **版本号用于失效，不用于逐请求强查询**  
   `perm_ver` 的作用是让权限变更可快速失效，而不是在每个请求中额外增加 Redis 读压力。

5. **允许轻微最终一致**  
   在权限变更即时性与鉴权性能之间做平衡。

## 3.3 Token 状态（status）设计说明

在本方案中，`auth:access:{token}`、`auth:refresh:{token}`、`auth:session:{sid}` 等对象均包含 `status` 字段：

```
ACTIVE / REVOKED / EXPIRED
```

该字段用于表达认证对象的**业务状态**，与 Redis 的 TTL（过期时间）形成互补关系。

---

### 3.3.1 为什么需要 status

Redis TTL 已经可以表达“是否过期”，但它仅能覆盖**时间驱动的失效**，无法表达**业务驱动的状态变化**。

引入 `status` 的核心原因如下：

#### （1）区分“自然过期”和“主动失效”

仅使用 TTL 时，以下场景在 Redis 中表现一致：

- token 正常过期
- 用户主动退出登录
- 管理员踢下线
- refresh token 被轮换
- 风控封禁

上述情况都会表现为 key 不存在，无法区分语义。

引入 `status` 后：

| 场景 | status |
|------|--------|
| 正常使用中 | ACTIVE |
| 主动退出 / 被踢下线 | REVOKED |
| 风控封禁 | REVOKED / BLOCKED（可扩展） |

---

#### （2）支持“未过期但立即失效”

很多安全相关场景要求：

> token 在 TTL 未到期前立即失效

例如：

- logout
- 踢下线
- refresh token rotation
- 风控策略触发

TTL 无法表达这一点，而 `status` 可以：

```
status = REVOKED
```

---

#### （3）提升系统可观测性与可审计性

使用 `status` 可以明确区分：

- token 是否被吊销
- 是否因安全策略失效
- 是否正常过期

这对于：

- 日志分析
- 审计系统
- 安全追踪

具有重要价值。

---

#### （4）降低分布式竞态问题

在分布式环境中，直接删除 key 可能产生竞态窗口：

```
请求 A：读取到 token（尚未删除）
请求 B：token 已被删除
```

使用 `status`：

```
status = REVOKED
```

可以保证所有节点读取到一致的业务状态，降低竞态带来的不一致问题。

---

### 3.3.2 status 与 TTL 的职责划分

| 机制 | 作用 |
|------|------|
| TTL | 控制 token 生命周期（自然过期） |
| status | 控制 token 业务状态（主动失效） |

可以理解为：

> TTL 决定“什么时候结束”，status 决定“当前还能不能用”。

---

### 3.3.3 状态变更时机

以下为各类典型场景下的 `status` 变更策略：

---

#### （1）登录成功

```
status = ACTIVE
```

适用于：

- access token
- refresh token
- session

---

#### （2）用户主动退出（Logout）

```
auth:access:{token}   -> REVOKED
auth:refresh:{token}  -> REVOKED
auth:session:{sid}    -> REVOKED
```

说明：

- 可选择**不立即删除 key**
- 推荐依赖 TTL 或异步清理

---

#### （3）管理员踢下线

与 logout 相同：

```
status = REVOKED
```

---

#### （4）Refresh Token 轮换（Rotation）

```
旧 refresh token -> REVOKED
新 refresh token -> ACTIVE
```

说明：

- 防止 refresh token 被重复使用（重放攻击）
- 强烈建议在生产环境启用

---

#### （5）权限变更

本方案默认策略：

- 不修改 status
- 通过 `perm_ver` + 缓存失效控制权限更新

可选增强策略（高安全场景）：

```
auth:session:{sid} -> REVOKED
```

强制用户重新登录

---

#### （6）风控封禁 / 账号冻结

```
status = REVOKED
```

或扩展状态：

```
status = BLOCKED
```

说明：

- 推荐保留 key 以便审计
- 不建议直接删除

---

#### （7）Token 自然过期

通常不需要显式设置：

```
status = EXPIRED
```

原因：

- TTL 已自动删除 key
- 每次鉴权会校验 `expireAt`

仅在有审计需求时可考虑记录 EXPIRED 状态。

---

### 3.3.4 是否需要立即删除 Key

不建议在状态变更时立即删除 key，推荐策略：

- 优先设置：

```
status = REVOKED
```

- 删除策略采用：

   - TTL 自然过期
   - 或异步清理任务

原因：

- 减少 Redis 写压力
- 提升系统稳定性
- 保留短期审计能力

---

### 3.3.5 对鉴权流程的影响

在资源请求鉴权中，仅需增加一步判断：

```java
if (!"ACTIVE".equals(status)) {
    throw new UnauthorizedException();
}
```

该判断：

- 不增加 Redis 查询次数
- 不影响热路径性能
- 显著增强安全可控性

---

### 3.3.6 小结

`status` 的设计目标不是替代 TTL，而是补充 TTL，使 token 生命周期具备业务语义：

> TTL 负责“自然过期”，status 负责“主动失效”。

在保证高性能（单次 Redis 读取）的前提下，引入 `status` 可以显著提升系统的：

- 安全性
- 可控性
- 可观测性
- 一致性表现

是中高并发认证系统中的推荐实践。

## 4. 适用范围与边界

### 4.1 适用范围

- 统一登录认证中心
- API 网关 + 后端微服务
- 中高 QPS 业务系统
- 需要踢下线、权限调整、会话管理的后台系统
- 不希望 JWT 直接暴露全部 claims 的系统

### 4.2 不在范围内

- 授权码模式
- 第三方社交登录
- ABAC / PBAC / Policy Engine 的完整设计
- 多租户隔离的复杂权限域设计

## 5. 总体方案

### 5.1 核心对象

- **Session**：一次登录态，代表用户在某个终端上的有效会话。
- **Access Token**：短期访问凭证，用于资源请求鉴权。
- **Refresh Token**：长期续期凭证，用于续签 access token。
- **Permissions Snapshot**：某一时刻的权限快照，通常与 `perm_ver` 绑定。
- **Permission Version（perm_ver）**：权限版本号，用于让旧权限快照失效。

### 5.2 方案结论

- **资源请求热路径**：只查 `auth:access:{token}`。
- **Session 信息**：仅用于登录、刷新、退出、踢下线等低频场景。
- **权限集合**：默认走本地缓存；必要时按 `uid + perm_ver` 维度加载。
- **权限变更**：通过 `perm_ver` 递增 + 缓存失效，确保旧权限不再被使用。
- **不在热路径中额外查 `auth:session:{sid}`、`auth:user:{uid}:perm_ver`、`auth:user:{uid}:perms`。**

### 5.3 关系图

```text
User 1 ---- N Session 1 ---- N Access Token
```

说明：

- 一个用户可以有多个 session（多端登录）。
- 一个 session 在生命周期内可以经历多次 access token 刷新。
- access token 是资源请求的唯一高频校验入口。
- session 是控制面管理入口，而不是每个请求都必须读取的对象。

## 6. Redis Key 设计

### 6.1 命名约定

统一采用如下命名空间：

- `auth:access:{token}`
- `auth:refresh:{token}`
- `auth:session:{sid}`
- `auth:user:{uid}:perms`
- `auth:user:{uid}:perm_ver`
- `auth:user:{uid}:sessions`

其中：

- `{token}` 建议为 token 原值或其哈希值。
- `{sid}` 为 session id。
- `{uid}` 为用户 id。

### 6.2 Key 列表

| Key | 作用 | 访问频率 | 是否在高频鉴权中读取 |
|---|---|---:|---|
| `auth:access:{token}` | access token 元数据 | 高频 | 是 |
| `auth:refresh:{token}` | refresh token 元数据 | 低频 | 否 |
| `auth:session:{sid}` | 登录会话信息 | 低频 | 否 |
| `auth:user:{uid}:perms` | 用户权限快照 | 中低频 | 默认否 |
| `auth:user:{uid}:perm_ver` | 权限版本号 | 低频 | 否 |
| `auth:user:{uid}:sessions` | 用户活跃 session 集合 | 低频 | 否 |

### 6.3 设计说明

#### 6.3.1 `auth:access:{token}` 是唯一高频入口

资源请求到达后，认证与授权的主入口就是 access token。该 key 中应包含完成一次请求所需的最小上下文：

- `uid`
- `sid`
- `clientId`
- `scope`
- `permVer`
- `status`
- `expireAt`

这样做的目的，是把鉴权热路径收敛成一次 Redis 查询。

#### 6.3.2 `auth:session:{sid}` 不参与热路径

Session 只用于：

- 登录后状态管理
- refresh token 续签校验
- 主动退出
- 踢下线
- 多端会话管理

#### 6.3.3 `auth:user:{uid}:perms` 与 `auth:user:{uid}:perm_ver` 作为控制面数据

这两个 key 主要用于：

- 登录时签发 token
- refresh 时重新计算权限快照
- 权限变更时更新权限状态
- 低频管理操作

它们不应该被放入每个请求的强依赖链路中。

## 7. 数据结构设计

### 7.1 `auth:access:{token}`

#### 用途

表示某个 opaque access token 的当前状态，是资源请求的唯一高频入口。

#### 推荐结构

```text
Key: auth:access:{token}
Type: Hash
TTL: 与 access token 一致
```

#### 字段设计

| 字段 | 类型 | 含义 |
|---|---|---|
| `token` | string | access token 原值或哈希值 |
| `uid` | string / long | 用户 id |
| `sid` | string | 所属 session id |
| `clientId` | string | 客户端标识 |
| `scope` | string | 作用域或权限摘要 |
| `permVer` | long | 签发时的权限版本 |
| `issuedAt` | long | 签发时间戳 |
| `expireAt` | long | 过期时间戳 |
| `status` | string | `ACTIVE` / `REVOKED` / `EXPIRED` |

#### 示例

```text
HSET auth:access:at_xxx
    token at_xxx
    uid 1001
    sid sid_12345
    clientId web
    scope "user:read,user:update,order:read"
    permVer 12
    issuedAt 1710000000
    expireAt 1710000900
    status ACTIVE
EXPIRE auth:access:at_xxx 900
```

---

### 7.2 `auth:session:{sid}`

#### 用途

表示一次登录会话的全局状态，仅在低频控制面使用。

#### 推荐结构

```text
Key: auth:session:{sid}
Type: Hash
TTL: 与 refresh token 一致
```

#### 字段设计

| 字段 | 类型 | 含义 |
|---|---|---|
| `sid` | string | session id |
| `uid` | string / long | 用户 id |
| `clientId` | string | 客户端标识 |
| `deviceId` | string | 设备标识 |
| `status` | string | `ACTIVE` / `REVOKED` / `EXPIRED` |
| `loginAt` | long | 登录时间戳 |
| `lastSeenAt` | long | 最近活跃时间戳 |
| `refreshToken` | string | 当前绑定的 refresh token 标识 |
| `accessToken` | string | 当前绑定的 access token 标识 |

#### 说明

- 该 key 的存在主要是为了管理登录态，而不是承载每次请求的鉴权成本。
- 退出登录、刷新 token、踢下线时需要读取它。
- 当需要强制失效某个 session 时，它是核心入口。

---

### 7.3 `auth:refresh:{token}`

#### 用途

表示 refresh token 的状态，用于续期 access token。

#### 推荐结构

```text
Key: auth:refresh:{token}
Type: Hash
TTL: 与 refresh token 一致
```

#### 字段设计

| 字段 | 类型 | 含义 |
|---|---|---|
| `token` | string | refresh token 原值或哈希值 |
| `uid` | string / long | 用户 id |
| `sid` | string | 所属 session id |
| `clientId` | string | 客户端标识 |
| `permVer` | long | 签发时权限版本 |
| `issuedAt` | long | 签发时间戳 |
| `expireAt` | long | 过期时间戳 |
| `status` | string | `ACTIVE` / `REVOKED` / `EXPIRED` |
| `rotatedFrom` | string | 被轮换前的旧 token，可选 |
| `rotatedTo` | string | 轮换后的新 token，可选 |

---

### 7.4 `auth:user:{uid}:perms`

#### 用途

保存用户的权限快照，用于登录、刷新、权限变更等控制面场景。

#### 推荐结构

```text
Key: auth:user:{uid}:perms
Type: String(JSON)
TTL: 可长可短，建议配合逻辑刷新
```

#### 示例

```json
{
  "uid": 1001,
  "roles": ["ADMIN", "OPS"],
  "permissions": [
    "user:read",
    "user:update",
    "order:read"
  ],
  "permVer": 1,
  "updatedAt": 1710000000
}
```

#### 使用方式

- 登录时读取一次，用于签发 access token 和 refresh token。
- refresh 时读取一次，用于生成新的权限快照。
- 权限变更时整体更新。
- **不建议每个资源请求都回源读取该 key。**

---

### 7.5 `auth:user:{uid}:perm_ver`

#### 用途

用于权限版本控制。

#### 规则

- 权限发生语义变化时，`perm_ver` 自增。
- access token 签发时携带 `permVer`。
- 新旧权限差异通过版本号体现。
- 版本号的变化主要用于触发缓存失效和控制面判断。

#### 说明

在本方案中，`perm_ver` 不作为每个请求必须回查 Redis 的对象。  
它更多用于：

- 权限变更后驱动失效
- 刷新 token 时重新签发
- 管理端做一致性控制

---

### 7.6 `auth:user:{uid}:sessions`

#### 用途

保存用户当前活跃的 session id 列表，用于多端管理、踢下线、全量注销。

#### 推荐结构

```text
Key: auth:user:{uid}:sessions
Type: Set
TTL: 可与最晚过期 session 保持一致
```

#### 示例

```text
SADD auth:user:1001:sessions sid_12345 sid_67890
```

## 8. 鉴权模型与性能策略

### 8.1 热路径约束

在高频资源请求中，推荐只做一次 Redis 读取：

```text
GET/HGETALL auth:access:{token}
```

然后在本地完成后续判断：

- access token 是否存在
- access token 是否有效
- 当前请求是否拥有目标权限
- 是否命中本地权限缓存

### 8.2 为什么不在热路径读取 `auth:session:{sid}`

因为 session 的存在性判断在大多数场景下属于低频控制面问题：

- 用户主动退出
- 管理员踢下线
- refresh token 续签
- 多端登录状态管理

这些场景不需要每个资源请求都实时读取 session。  
如果每次请求都查 session，会把原本“控制面”的成本强行注入“数据面”。

### 8.3 为什么不在热路径读取 `auth:user:{uid}:perm_ver`

因为版本号检查的本质是“失效通知”，不是“逐请求强校验”。

更合理的做法是：

- token 中携带 `permVer`
- 服务端本地缓存权限集合
- 权限变更时通过版本递增和缓存失效让旧权限自然淘汰

### 8.4 为什么不在热路径读取 `auth:user:{uid}:perms`

因为权限集合通常是较大的对象。  
它更适合用于：

- 本地缓存
- 登录 / 刷新时加载
- 变更时更新
- 管理端校验

而不是每次请求都直接从 Redis 拉取。

### 8.5 推荐的权限缓存策略

建议在认证中心或资源服务本地使用一层进程内缓存，例如：

- Caffeine
- Guava Cache
- 本地 LRU
- 短 TTL 缓存

缓存 key 推荐设计为：

```text
permissions:{uid}:{permVer}
```

#### 缓存命中规则

1. 先查 `auth:access:{token}` 获取 `uid`、`permVer`、`scope`。
2. 再查本地缓存 `permissions:{uid}:{permVer}`。
3. 命中则直接完成授权。
4. 未命中则由认证中心或控制面加载权限快照并回填缓存。

### 8.6 权限变更后的失效方式

当管理员修改用户权限时：

1. 数据库更新权限数据。
2. `auth:user:{uid}:perm_ver` 自增。
3. `auth:user:{uid}:perms` 更新。
4. 发布缓存失效事件，通知各节点清理旧的 `permissions:{uid}:{oldPermVer}`。

这样可以在不增加每请求 Redis 读压力的前提下，保证权限变更尽快生效。

## 9. 业务流程设计

### 9.1 登录流程

#### 目标

用户登录成功后签发 access token 与 refresh token，并建立 session。

#### 流程

1. 客户端提交用户名、密码、clientId、deviceId。
2. 认证服务校验用户名和密码。
3. 读取该用户的当前权限快照和 `perm_ver`。
4. 生成 `sid`、access token、refresh token。
5. 写入 `auth:session:{sid}`。
6. 写入 `auth:access:{token}`。
7. 写入 `auth:refresh:{token}`。
8. 更新 `auth:user:{uid}:sessions`。
9. 返回 token 给客户端。

#### 伪代码

```java
public LoginResponse login(String username, String password, String clientId, String deviceId) {
    User user = userRepo.findByUsername(username);
    assertPassword(password, user.getPasswordHash());

    String uid = String.valueOf(user.getId());
    long currentPermVer = redis.getLong("auth:user:" + uid + ":perm_ver");
    String permsJson = redis.get("auth:user:" + uid + ":perms");

    String sid = idGenerator.nextSid();
    String accessToken = tokenGenerator.newAccessToken();
    String refreshToken = tokenGenerator.newRefreshToken();

    redis.hmset("auth:session:" + sid, mapOf(
        "sid", sid,
        "uid", uid,
        "clientId", clientId,
        "deviceId", deviceId,
        "status", "ACTIVE",
        "loginAt", String.valueOf(now()),
        "lastSeenAt", String.valueOf(now()),
        "refreshTokenId", refreshToken,
        "accessTokenId", accessToken,
        "permVer", String.valueOf(currentPermVer)
    ));
    redis.expire("auth:session:" + sid, REFRESH_TTL_SECONDS);

    redis.hmset("auth:access:" + accessToken, mapOf(
        "token", accessToken,
        "uid", uid,
        "sid", sid,
        "clientId", clientId,
        "scope", extractScope(permsJson),
        "permVer", String.valueOf(currentPermVer),
        "issuedAt", String.valueOf(now()),
        "expireAt", String.valueOf(now() + ACCESS_TTL_SECONDS),
        "status", "ACTIVE"
    ));
    redis.expire("auth:access:" + accessToken, ACCESS_TTL_SECONDS);

    redis.hmset("auth:refresh:" + refreshToken, mapOf(
        "token", refreshToken,
        "uid", uid,
        "sid", sid,
        "clientId", clientId,
        "scope", extractScope(permsJson),
        "permVer", String.valueOf(currentPermVer),
        "issuedAt", String.valueOf(now()),
        "expireAt", String.valueOf(now() + REFRESH_TTL_SECONDS),
        "status", "ACTIVE"
    ));
    redis.expire("auth:refresh:" + refreshToken, REFRESH_TTL_SECONDS);

    redis.sadd("auth:user:" + uid + ":sessions", sid);

    return new LoginResponse(accessToken, refreshToken, ACCESS_TTL_SECONDS);
}
```

### 9.2 资源请求鉴权流程

#### 目标

每个资源请求都应以尽可能低的 Redis 成本完成认证和授权。

#### 推荐流程

1. 从请求头提取 access token。
2. 读取 `auth:access:{token}`。
3. 判断 token 是否存在、是否过期、是否被吊销。
4. 从 access token 中提取 `uid`、`sid`、`permVer`、`scope`。
5. 在本地缓存中查找 `permissions:{uid}:{permVer}`。
6. 若命中缓存，则直接执行权限判断。
7. 若未命中缓存，则从控制面补齐权限快照并回填本地缓存。
8. 完成鉴权，放行请求。

#### 伪代码

```java
public AuthContext authenticate(String accessToken, String requiredPermission) {
    Map<String, String> access = redis.hgetAll("auth:access:" + accessToken);
    if (access.isEmpty()) {
        throw new UnauthorizedException();
    }

    if (!"ACTIVE".equals(access.get("status"))) {
        throw new UnauthorizedException();
    }

    long expireAt = Long.parseLong(access.get("expireAt"));
    if (expireAt <= now()) {
        throw new UnauthorizedException();
    }

    String uid = access.get("uid");
    long permVer = Long.parseLong(access.get("permVer"));

    PermissionsSnapshot snapshot = permissionCache.get(uid + ":" + permVer, key -> {
        return permissionService.loadPermissions(uid, permVer);
    });

    if (!snapshot.contains(requiredPermission)) {
        throw new ForbiddenException();
    }

    return AuthContext.from(access, snapshot);
}
```

#### 说明

- 这个流程里，**Redis 热路径只出现一次**。
- session、perm_ver、perms 不在每个请求中独立读取。
- 如果权限快照需要更强实时性，可通过权限变更事件驱动本地缓存失效，而不是通过每请求多次 Redis 查询来保证。

### 9.3 refresh token 刷新流程

#### 目标

在 access token 过期后，允许客户端使用 refresh token 获取新的 access token。

#### 流程

1. 客户端提交 refresh token。
2. 查询 `auth:refresh:{token}`。
3. 校验 refresh token 是否有效。
4. 查询 `auth:session:{sid}`，确认 session 是否仍然有效。
5. 从控制面读取该用户当前的权限快照和 `perm_ver`。
6. 签发新的 access token。
7. 如启用 refresh rotation，则同时签发新的 refresh token。
8. 更新 session 中绑定的 token 信息。
9. 返回新的凭证。

#### 说明

refresh 是低频场景，因此允许多一次 Redis 读取。  
它不是资源请求热路径，不需要像鉴权链路一样极限压缩。

---

### 9.4 退出登录流程

#### 目标

用户主动退出后，当前 session 和相关 token 立即失效。

#### 流程

1. 从 access token 定位当前 session。
2. 删除 `auth:access:{token}`。
3. 删除 `auth:refresh:{token}`。
4. 删除 `auth:session:{sid}`。
5. 从 `auth:user:{uid}:sessions` 中移除该 sid。
6. 触发本地缓存失效。

#### 说明

退出登录属于低频控制面，不要求每个资源请求都承担额外成本。

---

### 9.5 权限变更流程

#### 目标

当用户权限发生变化时，旧权限应尽快失效，新权限应尽快生效。

#### 流程

1. 管理员修改数据库中的权限记录。
2. 更新 `auth:user:{uid}:perms`。
3. `auth:user:{uid}:perm_ver` 自增。
4. 发布权限变更事件。
5. 各节点清理旧的本地权限缓存。
6. 如业务要求更严格，可选择强制注销该用户所有 session。

#### 策略建议

- **常规企业系统**：权限变更后依靠版本递增 + 缓存失效即可。
- **安全要求更高的系统**：权限变更后强制踢下线所有 session。
- **权限变更非常频繁的系统**：建议缩短 access token TTL，并加强本地缓存失效机制。

---

### 9.6 踢下线流程

#### 目标

管理员或风控系统强制终止某个用户的一个或全部会话。

#### 两种模式

**模式 A：踢掉某个设备**

- 需要 `sid`
- 删除对应 session、access token、refresh token

**模式 B：踢掉该用户所有设备**

- 遍历 `auth:user:{uid}:sessions`
- 逐个删除对应的 session 和 token
- 清空 session 集合
- 清理本地缓存

#### 说明

踢下线是典型的低频控制面逻辑，不应出现在资源请求热路径中。

---

## 10. 推荐落地策略

### 10.1 最终建议

建议采用如下最终策略：

- **鉴权热路径只查 `auth:access:{token}` 一次**
- **session、refresh、permissions、perm_ver 仅用于低频控制面**
- **权限集合使用本地缓存承接**
- **权限变更通过版本递增和缓存失效生效**
- **踢下线、退出登录、refresh rotation 走控制面逻辑**

### 10.2 一句话总结

> 高频请求只做最少必要校验；低频状态管理放到控制面。  
> 这样既保留 opaque token 的可控性，又避免把 Redis 变成每个请求的性能瓶颈。

## 11. 接口职责边界

### 11.1 登录接口

负责：

- 校验用户名密码
- 读取权限快照
- 创建 session
- 签发 access token / refresh token

### 11.2 资源鉴权接口

负责：

- 校验 access token
- 构建请求上下文
- 本地权限判定

### 11.3 刷新接口

负责：

- 校验 refresh token
- 检查 session
- 重新签发 token

### 11.4 管理接口

负责：

- 权限变更
- 踢下线
- 会话清理
- 权限版本推进

---

## 12. 可选增强项

- **Token 哈希存储**：避免明文 token 直接落 Redis。
- **Refresh Token Rotation**：每次刷新后轮换 refresh token，降低泄露风险。
- **本地缓存失效事件**：通过消息队列或发布订阅广播权限变更。
- **双活/多活部署**：缓存失效事件需具备最终一致和重试能力。
- **风控字段扩展**：可加入 IP、UA、登录地区等用于异常检测。

---

## 13. 结论

本方案的核心取舍很明确：

- **热路径只做一次 Redis 查询**
- **控制面负责会话和权限生命周期**
- **权限变更靠版本号 + 缓存失效推进**
- **在性能、可控性和工程复杂度之间取得平衡**
