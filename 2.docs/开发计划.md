# Memora 项目开发计划

## 1. 计划概述

本文档依据《Memora 产品需求文档》与《Memora 软件设计文档》制定，旨在以**快速验证、持续演进**的方式落地系统。整体开发策略为**先后端再前端**，优先搭建可独立测试的后端能力，再通过前端界面完成端到端闭环，最终通过多轮迭代打磨产品体验。

整个项目划分为 **七个迭代（Sprint）**，每个迭代聚焦一组可交付的功能增量，确保每一阶段结束时都能产出可运行、可演示的系统。

## 2. 开发策略

- **后端优先**：先完成全部核心接口与业务逻辑，通过 REST API 工具（Postman / curl）进行独立验证。
- **前端跟进**：在后端主要模块完成后，集中开发前端页面，并通过 Mock 或真实接口完成联调。
- **模块并行**：基于架构设计的低耦合接口，认证、文件存储、解析调度等模块可相对独立开发，缩短关键路径。
- **快速闭环**：第一个可演示版本（MVP）必须在 **第5迭代结束** 时，支持“上传文档 → 问答 → 引用 → 反馈”全流程。
- **持续优化**：后续迭代重点完善解析扩展、混合检索、反馈优化权重、前端交互增强等。

## 5. 各迭代详细任务

### Sprint 0 —— 项目基础与环境搭建（1 周）

**目标**：准备好开发框架和本地运行环境，确保各组件可连通。

**任务列表**：
- 初始化 Spring Boot 项目，集成 Spring Web, MyBatis Plus, Spring AI, Redis, Milvus SDK, MinIO SDK。
- 设计项目包结构，按架构模块划分目录（auth, knowledgebase, document, filestorage, parser, vectorstore, aimodel, qa, feedback, history）。
- 编写 Docker Compose 编排文件，启动 MySQL、Redis、Milvus、MinIO、Nginx。
- 实现统一异常处理、统一响应格式（`Result<T>`）、基础日志切面。
- 创建数据库初始化脚本，生成所有核心表（按照 SDD 第 7 章）。
- 配置 Milvus 连接并创建初始 Collection（预留 schema）。
- 配置 MinIO bucket 初始化。
- 搭建前端 Vite + Vue3 项目骨架，集成路由、Pinia、Axios 基础封装。

**交付物**：可启动的后端空服务、运行中的基础设施、数据库表结构、前端空壳页面。

---

### Sprint 1 —— 认证授权与用户管理（1 周）

**目标**：实现安全、轻量的用户认证体系，为后续所有接口提供身份校验。

**任务列表**：
- 实现 `sys_user`、`sys_role`、`sys_permission`、`sys_user_role`、`sys_role_permission` 表的 MyBatis Plus 映射。
- 编写 `AuthService`：注册（密码 BCrypt 哈希）、登录（校验并生成 Token）。
- 实现 `TokenManager`：JWT 风格透明令牌签发、验证，Redis 存储 Token 状态，支持 Refresh Token 轮换。
- 编写认证拦截器，从请求头提取 Token 并设置用户上下文。
- 实现基础 RBAC 校验注解 `@RequirePermission`，通过 `PermissionEvaluator` 判断权限。
- 提供认证接口：`POST /api/auth/register`, `login`, `refresh`, `logout`, `GET /api/auth/me`。
- 编写单元测试（MockMvc 测试接口）。

**交付物**：可独立测试的用户注册、登录、Token 刷新、权限拦截功能。

---

### Sprint 2 —— 知识库与文件存储（2 周）

**目标**：搭建知识库管理骨架，建立文件去重存储体系，为文档导入提供底层支撑。

**任务详细**：
- **知识库模块**：
    - 实现 `KnowledgeBaseService`：增删改查、设置默认知识库、状态管理。
    - 实现 `KnowledgeBaseController`，提供全部 REST API。
    - 发布/监听 `KnowledgeBaseDeletedEvent`，用于后续级联清理（暂时只记录日志）。
- **文件存储模块**：
    - 实现 `FileStorageService`：上传至 MinIO、计算 MD5、创建/查询 `file_object` 记录。
    - 文件去重逻辑：同一用户同一知识库下相同 MD5 的文件，返回已有对象并提示。
    - 引用计数管理：`incrementRef`, `decrementRef`，当计数为 0 时可删除物理文件（延迟删除或标记清理）。
    - 监听文档删除事件，实现引用计数递减与物理文件清理。
    - 所有与 MinIO 交互封装在 `MinioFileStorageAdapter` 中，实现 `FileStorageAdapter` 接口。
- **集成测试**：上传文件，验证 MD5 去重、引用计数变化、删除后清理。

**交付物**：知识库管理接口完整可用，文件存储独立可测，支持文件去重和引用控制。

---

### Sprint 3 —— 文档管理与解析索引（3 周）

**目标**：打通“上传文档 → 解析 → 切片 → 向量写入”全链路（首版仅支持 txt/md）。

**任务详细**：
- **文档管理模块**：
    - 实现 `DocumentService`：创建文档记录（绑定文件对象）、删除（触发事件）、列表查询。
    - 实现分类/标签管理：`CategoryService`, `TagService`，支持树形分类和多标签绑定。
    - 编写控制器暴露 REST API（文档上传触发点）。
- **解析切分模块**：
    - 定义 `DocumentParser` 接口，实现 `TxtParser`, `MarkdownParser`（简单按标题或空行切分）。
    - 实现 `ParseOrchestrator`：接收解析任务，异步执行解析+切分，发布 `DocumentParsedEvent`（携带切片列表）。
    - 解析任务状态记录（`document_parse_task`），支持查询。
    - 解析失败记录原因，更新文档状态。
- **向量检索模块**：
    - 实现 `ChunkRepository`：持久化 `document_chunk` 记录。
    - 实现 `VectorStoreService`：监听 `DocumentParsedEvent`，调用 `EmbeddingClient` 向量化切片，存入 Milvus，同时保存 chunk 元数据。
    - 实现 `search` 方法：接收向量和 topK，返回相似度结果（含 chunk 信息）。
- **AI 模型适配模块**：
    - 配置 Spring AI，实现 `EmbeddingClient` 适配一种固定 Embedding 模型（如 text-embedding-ada-002 或开源模型）。
    - 预留 `ChatClient` 接口（暂时 Mock，以待下迭代）。
- **测试**：上传 txt/md 文件，查询文档解析状态，验证 Milvus 数据写入，通过 API 简单查询向量检索结果。

**交付物**：文档上传→解析→索引的全链路功能可用，向量检索可独立验证。

---

### Sprint 4 —— 智能问答与引用（2 周）

**目标**：实现基于知识库的 RAG 问答，支持多轮会话、引用展示。

**任务详细**：
- **历史记录模块**：
    - 实现 `HistoryService`：会话创建、消息追加、引用快照保存。
    - 保证数据表 `chat_session`, `chat_message`, `message_citation` 的正确填充。
    - 查询历史会话列表、会话内消息。
- **问答服务模块**：
    - 实现 `QaSessionService`：
        - `createSession`：初始化会话（关联知识库）。
        - `ask`：核心流程，组合以下步骤：
            1. 从历史模块加载最近 N 轮消息。
            2. 调用 `EmbeddingClient.embed` 计算问题向量。
            3. 调用 `VectorStoreService.search` 检索相关切片。
            4. 构建 Prompt（含上下文、切片内容）。
            5. 调用 `ChatClient.chat` 生成回答（实现一个具体模型适配，如 OpenAI 或 Ollama，通过接口切换）。
            6. 根据检索结果和生成答案构建引用（`message_citation`），包括高亮范围、片段文本。
            7. 保存用户消息、助手回答及引用到历史模块。
            8. 返回 `AnswerResponse`（含生成文本、引用列表）。
    - 支持 SSE 流式输出（如使用 Spring AI Streaming）。
- **API 端点**：按 SDD 暴露 `/api/chat/sessions`, `ask`, `messages` 等。
- **前端预留**：使用 Postman 完整模拟多轮对话，验证上下文记忆和引用结构。

**交付物**：完整 RAG 问答功能，多轮对话，返回引用信息，可通过 API 测试验证。

---

### Sprint 5 —— 反馈系统与数据清理（1 周）

**目标**：完成反馈记录与知识删除的完整级联清理，保证数据一致性。

**任务详细**：
- **反馈模块**：
    - 实现 `FeedbackService`：保存喜欢/不喜欢记录，查询反馈统计。
    - 提供 `POST /api/messages/{id}/feedback`。
    - 发布 `FeedbackSubmittedEvent`（为未来优化预留监听点）。
- **级联删除增强**：
    - 知识库删除事件触发：文档模块删除所有文档 → 向量检索模块删除 chunk 与 Milvus 向量 → 文件存储模块减少引用并清理零引用文件。
    - 文档删除事件：直接触发向量清理与文件引用递减。
    - 保证所有删除操作的事务性（本地事务 + 最终一致性通过重试或日志补偿）。
- **历史数据保护**：验证删除知识后，历史消息、引用快照完整保留，但原始链接状态标记为“已删除”。
- **测试**：执行删除操作，检查数据库、MinIO、Milvus 数据变化，确认历史记录不受影响。

**交付物**：反馈功能可用，安全的级联删除机制，历史记录与知识生命周期解耦。

---

### Sprint 6 —— 前端核心开发（3 周）

**目标**：搭建完整前端界面，与后端全部接口联调通过，实现用户可操作的全流程闭环。

**任务详细**：
- **项目结构与基础组件**：
    - 配置路由、状态管理（Pinia Store 按领域拆分）。
    - 实现 `AppLayout`, `Sidebar` 等布局组件。
    - 封装 Axios 拦截器，自动附加 Token、处理 401 刷新逻辑。
- **页面开发**（按优先级）：
    1. **登录/注册页**：表单校验，调用认证接口。
    2. **首页/仪表盘**：展示知识库概览、最近会话入口。
    3. **知识库管理页**：列表、创建/编辑弹窗、删除确认。
    4. **文档管理页**：文档列表，上传组件（支持 txt/md），查看解析状态，删除文档，分类/标签管理（树形展示、标签选择）。
    5. **问答页**：聊天窗口（`ChatWindow`, `ChatInput`, `MessageBubble`），支持多轮对话，会话列表切换，新建会话，流式回答展示。
    6. **引用与反馈**：回答气泡下方展示引用面板（`CitationPanel`），支持原文高亮；提供喜欢/不喜欢按钮（`FeedbackButtons`）。
    7. **历史记录页**：历史会话列表，查看历史消息，继续追问。
    8. **个人设置页**（简易版）：修改密码、个人信息。
- **状态管理**：各 Store 与后端接口对接，管理加载状态与错误处理。
- **Mock 数据替换**：逐步替换为真实 API 调用。
- **基础交互体验**：空状态、加载动画、错误提示、确认对话框。

**交付物**：可独立运行的前端应用，所有 MVP 页面功能实现，与后端完全连通。

---

### Sprint 7 —— 联调测试、优化与文档（2 周）

**目标**：确保系统稳定可靠，性能可接受，并产出部署与开发文档。

**任务详细**：
- **端到端测试**：
    - 编写关键业务流程测试用例（注册→创建知识库→上传文档→提问→反馈→删除）。
    - 边界条件测试：空文件、超大文件、非法 Token、知识库无内容问答、重复上传。
    - 性能测试：并行上传、多轮对话时延。
- **优化**：
    - 索引写入批处理优化（如使用批量 Embedding）。
    - 查询缓存（热门问题结果？视情况）。
    - 前端懒加载、虚拟滚动（会话消息列表）。
    - 日志完善，增加关键节点耗时记录。
- **部署**：
    - 编写生产环境 Docker Compose 文件（含所有服务）。
    - Nginx 配置（反向代理、静态资源）。
    - 环境变量提取，支持模型切换配置。
    - 编写 `README.md`：环境要求、启动步骤、API 文档链接。
- **文档整理**：
    - 架构设计文档最终版。
    - 数据库设计文档（ER 图）。
    - 开发者指南（如何添加新解析器、如何切换模型等）。

**交付物**：生产级部署方案，完整的测试报告，用户操作手册与开发者文档。

## 7. 风险与对策

| 风险             | 影响    | 应对措施                                       |
|----------------|-------|--------------------------------------------|
| AI 模型接口不稳定或延迟高 | 问答体验差 | 预留模型适配层，可快速切换本地模型；实现超时重试与降级应答              |
| 文档解析误切分导致回答不准  | 准确性降低 | 首版仅支持结构化好的 txt/md，后续优化切分策略和混合检索            |
| Milvus 单机资源消耗大 | 部署受限  | 使用轻量级向量数据库替代方案（如 Chroma、Qdrant）作为 Fallback |
| 前后端接口联调耗时超出预期  | 延期    | 提前定义 Swagger API 文档，开发期间前端使用 Mock 并行       |
| 个人开发时间碎片化      | 进度延误  | 按迭代交付增量，优先级刚性，可适当砍掉非核心分类/标签功能延后            |

## 8. 附录：任务依赖关系

- Sprint 0 无依赖，必须最先完成。
- Sprint 1 可独立进行（仅依赖数据库）。
- Sprint 2 依赖 Sprint 1（认证拦截器），可先行开发文件存储模块。
- Sprint 3 严格依赖 Sprint 2（文件对象、文档记录），部分依赖 Sprint 0 的 Milvus 环境。
- Sprint 4 依赖 Sprint 3（向量检索与 Chunk 数据）与 Sprint 2（知识库）。
- Sprint 5 依赖 Sprint 4（反馈基于消息），删除清理贯穿多模块。
- Sprint 6 依赖 Sprint 4 与 5 的接口完成（问答与反馈），可与后端并行开发 Mock，但最终联调需等后端就绪。
- Sprint 7 依赖所有模块部署完成。

通过该计划，Memora 将以稳健、可验证的步伐逐步成型，最终达成“越用越懂你”的个人 AI 知识库系统。
