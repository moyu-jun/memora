# Memora 软件设计文档（SDD）

## 1. 文档概述

### 1.1 文档目的
本文档用于说明 Memora 的软件设计方案，覆盖系统架构、模块划分、数据设计、接口设计、部署方案及扩展设计，为后续开发实现提供统一依据。

### 1.2 产品定位
Memora 是一款面向个人用户的 AI 个人知识库问答系统。用户可通过上传文档与文本构建私有知识库，并以自然语言进行检索问答。系统支持多轮对话、引用高亮、知识反馈优化、知识库管理与历史记录留存。

### 1.3 本文档范围
- 前后端分离架构
- 前端：Vue3 + TypeScript
- 后端：Java 21 + Spring Boot 3.5 + Spring AI + MySQL + Redis + Milvus + MinIO + MyBatis Plus
- 后端为单体应用
- 单机部署，最终部署到 Debian 12 Linux 服务器
- 不考虑分布式微服务场景
- 不设计导出功能
- 不设计文档版本管理功能

---

## 2. 设计目标与约束

### 2.1 设计目标
1. 支持个人用户管理多个知识库。
2. 支持文档、文本导入，并形成可检索知识。
3. 支持多轮对话与上下文记忆。
4. 支持喜欢 / 不喜欢反馈，形成低耦合优化闭环。
5. 支持引用高亮与答案溯源。
6. 支持删除知识时清理原始数据、向量数据与关联数据。
7. 保留历史对话与问答记录，不因知识删除而消失。

### 2.2 设计约束
1. 后端必须保持单体结构，避免过度拆分。
2. 安全认证不使用 Spring Security 框架，采用自研轻量级认证组件。
3. 第一阶段实际实现仅支持 txt 与 markdown 的基础结构解析。
4. 架构层面必须预留 PDF、Word、表格、代码块、图片等扩展能力。
5. 向量检索、关键词检索、反馈优化均需以低耦合方式设计，方便后期演进。
6. 模型暂时固定，但必须预留替换与扩展接口。

## 3. 总体架构设计

### 3.1 架构风格
采用“前后端分离 + 后端单体应用 + 领域模块化”的设计方式。

- 前端负责交互、展示与状态管理。
- 后端提供统一 REST API。
- 后端内部以模块化方式组织业务，保证高内聚、低耦合。
- 运行形态为单机部署，不依赖分布式组件。

### 3.2 系统逻辑架构

```text
[ Vue3 + TS 前端 ]
        |
        | REST API
        v
[ 单体后端 Spring Boot 3.5 ]
        |
        +--> [ 认证与 RBAC ]
        +--> [ 知识库管理 ]
        +--> [ 文档管理与解析 ]
        +--> [ 检索与问答 ]
        +--> [ 反馈系统 ]
        +--> [ 历史记录 ]
        +--> [ 标签 / 分类管理 ]
        +--> [ 文件存储适配 ]
        +--> [ 向量检索适配 ]
        +--> [ Spring AI 模型适配 ]
        |
        +--> MySQL
        +--> Redis
        +--> Milvus
        +--> MinIO
```

### 3.3 部署架构
- 本地开发与测试：开发机
- 生产部署：Debian 12 Linux 服务器
- 容器化部署：Docker / Docker Compose
- 不采用 Kubernetes
- 不拆分微服务

建议部署组件：
- `memora-web`：前端静态资源
- `memora-api`：后端单体服务
- `mysql`
- `redis`
- `milvus`
- `minio`
- `nginx`：反向代理与静态资源入口（推荐）

## 6. 核心业务设计

### 6.1 认证与 RBAC 设计

#### 6.1.1 认证方式
- Access Token：短期有效，用于请求认证
- Refresh Token：长期有效，用于刷新 Access Token
- Token 均为透明令牌
- 认证由自研轻量级组件完成

#### 6.1.2 RBAC 模型
采用最基础 RBAC：
- 用户 User
- 角色 Role
- 权限 Permission
- 用户-角色关联
- 角色-权限关联

#### 6.1.3 基础认证接口
- 注册
- 登录
- 刷新 token
- 登出
- 获取当前用户信息

#### 6.1.4 安全建议
- 密码单向加密存储
- Refresh Token 可做轮换
- Token 失效状态写入 Redis
- 所有认证信息走统一拦截链路

---

### 6.2 知识库设计

#### 6.2.1 知识库模型
支持多知识库，一个用户可拥有多个知识库。

#### 6.2.2 知识库能力
- 新增
- 编辑
- 删除
- 查询
- 设置默认知识库
- 按状态启停

#### 6.2.3 知识库使用方式
文档、标签、分类、问答、历史记录都围绕知识库进行归属与查询。

---

### 6.3 文件存储设计

#### 6.3.1 方案选择
采用 **MinIO 对象存储 + MySQL 元数据管理** 的方案。

原因：
- 单机部署友好
- 支持后期扩展
- 便于文件与业务解耦
- 便于做 md5 去重与引用统计

#### 6.3.2 文件对象管理策略
建立独立的文件对象记录，保存：
- 原始文件名
- 文件大小
- MIME 类型
- MD5
- MinIO 存储 Key
- 引用计数
- 上传人
- 创建时间

#### 6.3.3 重复文件处理策略
同一用户上传同一知识库中的相同 MD5 文件时：
1. 系统提示“文件已上传”。
2. 用户可选择：
    - 直接取消
    - 复用已有文件并重新绑定当前知识库内容
    - 删除旧文档后重新上传新内容
3. 不提供文档版本管理。
4. 物理文件按 MD5 去重保存，引用计数为 0 时删除对象。

说明：
- 这不是版本管理。
- 这是“内容去重 + 引用控制”的存储策略。

### 6.4 文档解析设计

#### 6.4.1 支持格式
设计层面支持：
- PDF
- Word
- Markdown
- TXT

#### 6.4.2 实际第一阶段实现
第一阶段仅实现：
- TXT
- Markdown
- 基础结构解析
- 不解析图片

#### 6.4.3 解析扩展能力
架构上预留：
- PDF 解析器
- Word 解析器
- 表格解析器
- 代码块解析器
- 图片 / OCR 解析器

#### 6.4.4 解析器设计原则
采用解析器接口 + 工厂模式：
- `DocumentParser`
- `TxtParser`
- `MarkdownParser`
- `PdfParser`
- `WordParser`
- `TableExtractor`
- `CodeBlockExtractor`
- `ImageExtractor`

首版仅注册 TXT / Markdown 实现，其余为预留实现点。

#### 6.4.5 切分策略
建议按层级逐步演进：
1. 基础结构切分
2. 段落切分
3. 语义切分
4. 混合切分

首版使用基础结构切分，保证稳定性与易实现性。

---

### 6.5 向量检索设计

#### 6.5.1 选择方案
采用 Milvus 作为向量数据库。

#### 6.5.2 设计原则
- 检索与业务解耦
- 查询能力通过适配器封装
- 首版实现基础向量检索
- 关键词检索接口预留
- 后续可扩展混合检索与重排序

#### 6.5.3 Collection 策略
建议采用单一 collection 按 embedding 模型版本管理。
- 当前固定模型对应一个 collection
- 后续若切换 embedding 模型，可新增 collection
- 业务侧通过检索适配器屏蔽细节

#### 6.5.4 检索结果输出
检索结果需返回：
- chunk_id
- 文档 id
- 知识库 id
- 相似度分数
- 来源定位信息
- 可用于高亮的片段信息

---

### 6.6 问答与多轮对话设计

#### 6.6.1 对话模式
支持多轮对话，具备上下文记忆。

#### 6.6.2 上下文构建方式
每次提问时，系统可使用：
- 当前问题
- 当前会话摘要
- 最近若干轮对话
- 检索到的知识片段

#### 6.6.3 记忆策略
建议采用：
- 会话级摘要
- 最近消息窗口
- 检索上下文拼接

这样既能保留上下文，又能避免 prompt 过长。

#### 6.6.4 回答生成原则
- 优先使用知识库内容回答
- 不足时明确说明“未找到相关依据”
- 回答应尽量结构化
- 回答必须可追溯

---

### 6.7 引用高亮设计

#### 6.7.1 引用目标
答案中必须展示知识来源，提升可信度。

#### 6.7.2 高亮设计
对每个引用记录保存：
- 来源文档
- 来源切片
- 片段文本
- 命中位置
- 高亮范围
- 来源定位信息

#### 6.7.3 第一阶段实现方式
- 对 txt / markdown 保存文本偏移范围
- 前端根据偏移范围高亮展示
- 预留 PDF 页码、Word 段落、表格块等定位字段

#### 6.7.4 后续扩展
未来可支持：
- PDF 页码定位
- 表格单元格定位
- 代码块定位
- 图片块定位

---

### 6.8 反馈系统设计

#### 6.8.1 反馈形式
- 喜欢
- 不喜欢

#### 6.8.2 设计原则
- 简单版本
- 低耦合
- 高内聚
- 方便后期扩展

#### 6.8.3 反馈作用
反馈不直接替代回答生成，而是作为优化信号，主要用于：
- 调整检索优先级
- 调整排序策略
- 调整上下文构建策略
- 调整回答风格偏好

#### 6.8.4 反馈记录建议
记录以下信息：
- 所属回答
- 所属问题
- 反馈类型
- 反馈时间
- 反馈用户
- 当前上下文快照

---

### 6.9 历史记录设计

#### 6.9.1 记录内容
- 会话历史
- 消息历史
- 问答历史
- 反馈历史

#### 6.9.2 历史保留原则
删除知识时：
- 删除原始文档
- 删除切片
- 删除向量
- 删除文件对象引用（若无其他引用）
- 删除知识库相关关联数据

但不删除：
- 历史会话
- 历史问答
- 历史反馈

#### 6.9.3 历史记录与删除知识的关系
历史记录应尽量保存回答快照与引用快照。
当原始知识被删除时：
- 历史回答仍保留
- 引用可显示“原始来源已删除”
- 仍可查看当时的回答结果

---

### 6.10 标签与分类设计

#### 6.10.1 设计目标
帮助用户从多个维度管理知识。

#### 6.10.2 设计方案
采用“分类 + 标签”双体系：

- **分类**：用于主归属，支持树形结构
- **标签**：用于交叉标记，支持多选

#### 6.10.3 使用方式
- 一个文档可归属一个主分类
- 一个文档可绑定多个标签
- 分类和标签均属于某个知识库

#### 6.10.4 好处
- 分类便于结构化管理
- 标签便于检索与聚合
- 适合个人知识库的灵活整理方式

## 9. 关键业务流程

### 9.1 登录与刷新流程
1. 用户登录。
2. 后端校验账号密码。
3. 返回 access token 与 refresh token。
4. access token 用于请求认证。
5. access token 过期后通过 refresh token 刷新。
6. 用户登出后 token 失效。

### 9.2 文档上传与解析流程
1. 用户选择知识库并上传文件。
2. 系统计算 MD5。
3. 若当前知识库内已存在同 MD5 文件，提示重复上传。
4. 用户选择是否继续复用或取消。
5. 文件写入 MinIO。
6. 创建文档记录。
7. 创建解析任务。
8. 解析、切分、索引、入库。
9. 更新文档状态为可问答。

### 9.3 问答流程
1. 用户输入问题并选择知识库。
2. 系统读取当前会话上下文。
3. 检索相关切片。
4. 构建上下文。
5. 调用大模型生成答案。
6. 保存回答、引用与上下文快照。
7. 返回前端展示。

### 9.4 反馈流程
1. 用户对 assistant 回复点击喜欢 / 不喜欢。
2. 系统保存反馈记录。
3. 后续问答在检索、排序、上下文构建中参考反馈信号。

### 9.5 删除知识流程
1. 用户删除知识库或文档。
2. 系统删除关联文档、切片、向量、标签关系、分类关系、文件引用。
3. 若文件对象引用计数为 0，则删除 MinIO 原文件。
4. 历史会话、消息、反馈保留。
5. 历史引用保留快照信息。

## 10. 业务规则

1. 文档删除必须同步清理向量数据。
2. 删除知识不删除历史对话数据。
3. 历史回答保留引用快照。
4. 文件去重以 MD5 为基础。
5. 文档不做版本管理。
6. 知识库支持新增、编辑、删除。
7. 问答必须支持多轮上下文。
8. 反馈仅为喜欢 / 不喜欢。
9. 引用必须可高亮。
10. 第一阶段只实现 txt 与 markdown 基础结构解析。

## 13. 关键设计取舍

### 13.1 为什么后端采用单体
- 项目处于产品早期
- 核心复杂度在 RAG 与知识组织，不在服务拆分
- 单体更利于快速迭代与调试
- 便于个人项目持续演进

### 13.2 为什么使用 MinIO
- 单机部署友好
- 与业务解耦
- 支持文件去重与引用控制
- 后续迁移成本低

### 13.3 为什么使用 Milvus
- 适合向量检索主链路
- 可独立适配模型与检索策略
- 后续可扩展多模型与多策略

### 13.4 为什么反馈设计为轻量级
- 首版先验证闭环
- 避免过早复杂化
- 方便后续扩展更精细的优化策略

## 14. 风险与应对

### 14.1 检索不准确
应对：
- 先保证结构化切分质量
- 后续再加入混合检索与重排序

### 14.2 多轮上下文膨胀
应对：
- 使用会话摘要
- 保留最近窗口
- 控制上下文长度

### 14.3 解析扩展复杂
应对：
- 使用统一解析器接口
- 通过工厂与适配器解耦
- 首版只落地 txt / markdown

### 14.4 删除知识后历史数据关联问题
应对：
- 引用快照持久化
- 历史回答与原始知识解绑
- 原始知识硬删除，历史保留

### 14.5 反馈数据过少
应对：
- 反馈作为增强信号，不作为唯一依据
- 保持默认检索与生成能力稳定

## 16. 结语

Memora 的设计目标不是简单做一个“能回答问题的工具”，而是做成一个能够持续积累、持续优化、可长期使用的个人知识系统。

本设计采用单体架构、前后端分离、单机部署的方式，优先保证：
- 可落地
- 可维护
- 可扩展
- 可演进

在第一阶段，通过最小可用闭环完成真实验证；在后续阶段，再逐步引入更强的解析、检索、优化与个性化能力。
